## Problem Statement

Engineers running multiple AI coding agents in parallel (Claude Code, Codex, Cursor CLI) lack a unified, visually intuitive way to monitor all sessions at a glance. Existing tools are either good terminal multiplexers (cmux, claude-squad, dmux) or good visualizations (AgentOffice, Bit Office, Pixel Agents), but none combine real terminal sessions with a delightful visual awareness layer. The user wants a tool that makes managing multiple AI terminal sessions fun — not just efficient.

## What Makes This Cool

The core delight is **seeing your AI workforce**. Instead of alt-tabbing between terminal tabs, you glance at a pixel-art virtual office and instantly know: which agents are working (typing animation), which need your attention (message bubble), and which are idle. Clicking a character switches to their terminal. The office metaphor turns an abstract coordination problem into something spatial and intuitive.

The "whoa" moment: five pixel agents at desks, keyboards clacking, one stands up with a speech bubble — you click it, answer the question, and the character sits back down. You're a calm manager watching through a glass wall, not a frantic alt-tabber.

## Constraints

- Side project — built alongside main startup work
- Two-person eng team (user on Claude Code, colleague on Cursor)
- Must not compromise the terminal experience — xterm.js must feel as good as native terminal
- Single-player first, multiplayer-ready architecture
- Browser-based delivery (localhost web app)
- Agent-agnostic — any CLI tool that runs in a terminal should work (Claude Code, Codex, Cursor CLI, etc.)

## Premises

1. **The primary value is delight (fun UX), not productivity.** The user has Conductor and it works. Daegu exists because managing agents should be more fun.
2. **Single-player is the core product; multiplayer is a nice-to-have for later.** The user and colleague don't coordinate much today — they merge at the end. Multiplayer adds value but isn't the reason to build this.
3. **Browser-based with xterm.js is the right delivery format.** Enables future multiplayer (share a URL), avoids native app complexity, and xterm.js provides a real terminal experience.
4. **Must connect to REAL terminal sessions to be a daily-driver.** AgentOffice is a toy because it runs simulated agents. Daegu's terminal side must be production-quality — real PTY sessions, real Claude Code.
5. **This is a side project alongside the main startup work.** Scope accordingly — aim for a delightful MVP, not a platform.

## Approaches Considered

### Approach A: Full-Stack Pixel Terminal (CHOSEN)
Browser-based app with pixel office (Pixi.js/canvas) left panel + xterm.js terminal right panel, backed by a Node.js server with node-pty for real PTY sessions. WebSocket bridges everything.

- **Effort:** M (human: ~2 weeks / CC: ~6-10 hours — ~2-3 hours for core plumbing, additional time for pixel art integration, animation polish, and state detection tuning)
- **Risk:** Low — all pieces are battle-tested
- **Pros:** Full control, real terminal, multiplayer-ready, daily-driver from day one
- **Cons:** Most code upfront, pixel art assets needed

### Approach B: Minimal Terminal Manager + Overlay
Ship terminal management first (xterm.js + node-pty), add pixel art as a thin overlay later. Initially just colored dots or icons for agent state.

- **Effort:** S (human: ~3-4 days / CC: ~1 hour)
- **Risk:** Low
- **Pros:** Usable fast, proves terminal works first
- **Cons:** No "whoa" factor — the delight IS the product, shipping without it misses the point

### Approach C: Fork & Remix AgentOffice
Fork AgentOffice's pixel rendering, replace autonomous agent backend with xterm.js + node-pty terminal management.

- **Effort:** S-M (human: ~1 week / CC: ~1-2 hours)
- **Risk:** Medium — inheriting unknown code quality
- **Pros:** Pixel art rendering for free, faster path to visuals
- **Cons:** Dependency on someone else's code, may not match vision

## Recommended Approach

**Approach A: Full-Stack Pixel Terminal.** The primary value of daegu is delight. Approach B defers the delight. Approach C ties you to someone else's vision. With CC+gstack, the effort difference between A and B is hours, not weeks. Build the real thing.

### Architecture

```
Browser (localhost:7007)
├── Left Panel: Pixel Office (Pixi.js / HTML5 Canvas)
│   ├── Floor grid with pixel-art aesthetic
│   ├── Desks (auto-positioned in 2-column grid layout)
│   ├── Agent characters (color-coded, animated)
│   │   ├── Typing animation → WORKING state
│   │   ├── Message bubble → MESSAGE state
│   │   ├── Idle pose → IDLE state
│   │   └── Red X overlay → DISCONNECTED state
│   └── "+" button → spawn new agent (see Agent Spawn UX below)
│
├── Right Panel: xterm.js Terminal
│   ├── Tab bar (one tab per agent, status dots)
│   ├── Terminal viewport (connected to PTY via WebSocket)
│   └── xterm.js fit addon handles resize → sends resize event via WebSocket
│
└── Top Bar: agent count, attention count, settings

Server (Node.js, localhost:7007)
├── HTTP: serves frontend static files
├── WebSocket (framed JSON protocol):
│   ├── Client → Server:
│   │   ├── {type: "pty-input", sessionId, data}    — keyboard input
│   │   ├── {type: "pty-resize", sessionId, cols, rows} — terminal resize
│   │   ├── {type: "spawn", command, name?}          — create new session
│   │   └── {type: "kill", sessionId}                — terminate session
│   ├── Server → Client:
│   │   ├── {type: "pty-output", sessionId, data}    — terminal output (base64)
│   │   ├── {type: "state-change", sessionId, state} — agent state update
│   │   ├── {type: "session-created", sessionId, name, color} — new session info
│   │   └── {type: "session-ended", sessionId, reason} — session terminated
│   └── All messages are JSON with a "type" discriminator field.
│       PTY output data is base64-encoded to safely embed binary terminal
│       data (ANSI escape sequences, etc.) within JSON.
│
├── PTY Manager: spawns/manages node-pty processes
│   ├── spawn(command, name?) → creates new PTY session
│   ├── write(sessionId, data) → sends input to PTY
│   ├── resize(sessionId, cols, rows) → sends SIGWINCH to PTY
│   ├── onData(sessionId, callback) → receives PTY output
│   └── kill(sessionId) → terminates PTY session
│
├── State Detector (server-side, parses raw PTY output):
│   │ NOTE: raw PTY output is ANSI-stripped before regex matching
│   │ (strip all escape sequences: /\x1b\[[0-9;]*[a-zA-Z]/g and
│   │  control chars: /[\x00-\x1f\x7f]/g except \n\r\t)
│   │
│   ├── WORKING → PTY output received in the last 3 seconds
│   ├── IDLE → no PTY output for 3+ seconds AND no prompt detected
│   ├── WAITING → agent prompt regex matched as last line,
│   │            no output for 3+ seconds. Claude Code prompts:
│   │            - /^❯\s*$/  (main prompt)
│   │            - /^>\s*$/   (continuation prompt)
│   │            - /\$ $/     (shell prompt after exit)
│   ├── MESSAGE → output contains approval/question pattern:
│   │            - /Do you want to proceed\?/i
│   │            - /\(y\/n\)/i
│   │            - /\[Y\/n\]/i
│   │            - /Allow .+ to (read|edit|write|execute)/i
│   │            - /Press Enter to continue/i
│   └── DISCONNECTED → PTY process exited (exit event from node-pty)
│   │
│   └── State changes emitted to client via WebSocket event.
│       False positives acceptable in MVP. Heuristics are configurable
│       per agent type (stored in a patterns config file).
│
├── Session Persistence: PTY processes run independent of browser
│   ├── Server keeps all PTY sessions alive until explicitly killed
│   ├── Browser disconnect → sessions continue running
│   ├── Browser reconnect → reattaches to existing sessions,
│   │   replays from ring buffer (last 5000 lines per session;
│   │   oldest lines dropped when buffer is full)
│   └── Server restart → sessions lost (no tmux layer in MVP;
│       add optional tmux backing in v2 for full persistence)
│
└── Agent Metadata: stored in server-side session objects
    ├── name: user-editable, auto-generated if not specified
    │         (e.g., "agent-1", "agent-2", or based on command name)
    ├── color: auto-assigned from a 10-color palette, cycling on creation
    ├── command: the shell command this session runs
    └── createdAt: timestamp
```

**Runtime decision: Node.js** (not Bun). node-pty requires native N-API bindings. Bun's compatibility with node-pty is unverified and risky for a core dependency. Node.js is the safe choice.

**Terminal resize handling:** xterm.js fit addon detects panel resize → sends `pty-resize` message to server via WebSocket → server calls node-pty resize() → PTY process receives SIGWINCH. Required for the split-panel layout where users may drag the divider.

### Agent Spawn UX

When user clicks "+":
1. A minimal inline form appears with:
   - **Command** input field, pre-filled with `claude` (the default)
   - **Name** input field, pre-filled with auto-generated name (e.g., "agent-3")
   - **Start** button
2. User can accept defaults (just hit Enter/Start) or customize the command (e.g., `codex`, `cursor`, or any CLI tool).
3. On start: server spawns PTY with the specified command, a new character walks to an empty desk in the office, and a new tab appears in the terminal panel.

### Pixel Art Assets (MVP Decision)

For MVP: use a free pixel art character asset pack from itch.io (e.g., "Tiny RPG" style packs with idle/walk/sit animations). This unblocks implementation immediately. Custom sprites can replace them later once the visual identity is established. Characters are recolored per-agent using canvas tinting.

### Failure Modes

- **PTY process crashes:** Character shows DISCONNECTED state (red X overlay). User can click to see exit code/error in terminal, then dismiss or restart.
- **WebSocket disconnects:** Client auto-reconnects with exponential backoff (1s, 2s, 4s, max 30s). During disconnect, office shows a subtle "reconnecting..." banner. On reconnect, replays buffered output from ring buffer.
- **Server restart:** All PTY sessions are lost (MVP limitation). On browser reconnect, office shows empty with a message: "Server restarted — sessions ended. Click + to start new agents."
- **xterm.js rendering issues:** Fallback to canvas renderer if WebGL renderer fails (xterm.js supports both backends).

### Security Note

Daegu binds to `localhost` only. The server exposes full PTY access (equivalent to a shell) and must never be exposed to a network. This is intentional — it is a local development tool. The future multiplayer feature will require a separate auth/trust model (SSH tunneling, token auth, or similar) and will be designed when that feature is scoped.

## Open Questions

1. **Office layout algorithm:** Fixed desk positions (2-column grid) for MVP. Auto-layout with spatial logic (agents walk to desks, desks rearrange) deferred to v2.
2. **Agent-specific state detection:** Claude Code heuristics are defined above. Codex, Cursor CLI, and other agents will need their own prompt regex patterns added to the patterns config. For MVP, ship with Claude Code detection only; other agents fall back to timing-based WORKING/IDLE detection.
3. **Keyboard shortcut navigation:** Should there be hotkeys to switch between agents (Cmd+1, Cmd+2, etc.)? Likely yes for power users, but not in MVP scope.

## Success Criteria

- Can spawn a Claude Code instance from the browser and interact with it through xterm.js with no noticeable difference from native terminal
- Agent characters in the pixel office animate in response to real terminal state (typing when working, bubble when message)
- Clicking a character switches the terminal panel to that agent's session
- Can run 3+ agents simultaneously without performance issues
- Sessions survive browser tab close and reconnect (server keeps PTYs alive)
- The reaction when showing someone is "that's sick" not "that's useful"

## Reviewer Concerns

Minor issues identified in round 3 adversarial review (quality score: 8/10). None require structural changes — addressable during implementation:

1. **ANSI stripping completeness:** The ANSI strip regex only covers CSI sequences. Claude Code emits OSC sequences (terminal titles, hyperlinks) that will bypass the regex. Fix at implementation time: use the `strip-ansi` npm package or an expanded regex covering OSC/DCS sequences.

2. **IDLE vs. WAITING boundary:** "Last line" concept is underspecified. Define at implementation time as: "the last non-empty line after ANSI stripping from the most recent output burst, where a burst ends after 500ms of silence."

3. **State-to-color mapping:** Add to implementation: WORKING=yellow, WAITING=green, MESSAGE=orange (attention), IDLE=gray, DISCONNECTED=red.

4. **Ring buffer replay throttling:** Replaying 5000 lines per session on reconnect may freeze xterm.js. Throttle replay into chunks (500 lines per 50ms) or lazy-load per session on tab switch.
