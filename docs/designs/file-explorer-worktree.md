## Detailed Specs

### Worktree Lifecycle

**Path convention:** `~/.agent-007/worktrees/{repo-slug}/{branch-name}/`
- `repo-slug` is derived from the repo's absolute path via hash:
  `slugify(basename(path)) + '-' + shortHash(path)` (e.g., `myapp-a3f2`).
  This prevents collisions when two repos share the same basename
  (e.g., `/Users/bill/myapp` and `/Users/alice/myapp`).
- `branch-name` is auto-generated: `agent-007/{agent-name}` (e.g., `agent-007/agent-3`)
- The persisted config stores the user-visible `slug` as the basename for display,
  and uses `repoPath` (absolute) as the authoritative identifier. The `repo-slug` in
  the worktree path is derived, never stored.

**Creation:** At agent spawn time, after repo is selected in spawn form:
1. Server validates the path is a git repo: `git -C {repoPath} rev-parse --git-dir`
   - If it fails: return error "Not a git repository"
   - If the repo has no commits (`git rev-parse HEAD` fails): return error
     "Repository has no commits — make an initial commit first"
2. Server runs `git -C {repoPath} worktree add {worktreePath} -b {branchName}`
3. If the branch already exists (exit code non-zero, stderr contains "already exists"):
   return error "Branch name already in use — choose a different agent name"
4. PTY process is spawned with `cwd: worktreePath` (instead of `process.env.HOME`)

**Session schema changes:** Add to the session object:
```
repoPath: string | null     // absolute path to the git repo
worktreePath: string | null  // absolute path to the worktree directory
branchName: string | null    // git branch name
repoSlug: string | null      // human-readable repo display name (basename)
fileTree: object | null      // latest scanned file tree
changedCount: number         // count of changed files
conflicts: string[]          // file paths conflicting with other agents
```

**spawn WebSocket message change:**
```json
{ "type": "spawn", "command": "claude", "name": "agent-3", "repoPath": "/Users/bill/myapp" }
```

**session-created broadcast change:**
```json
{ "type": "session-created", "sessionId": "...", "name": "...", "color": "...",
  "command": "...", "state": "...", "repoSlug": "myapp", "branchName": "agent-007/agent-3",
  "changedCount": 0, "conflicts": [] }
```

**Known constraint: same-branch collision.** A git branch can only be checked out in
one worktree at a time. If user tries to create two agents on the same branch, the
second `git worktree add` will fail. This is handled by auto-generating unique branch
names (`agent-007/{agent-name}`). If a name collision occurs (user manually names two
agents the same thing), the spawn form shows an error.

**Known constraint: non-git directories.** If the user provides a path that is not a git
repo, validation step 1 catches it and returns a clear error message.

**Known constraint: no-commit repos.** Repos with zero commits have no HEAD, so
`git worktree add` fails. Validation step 1 catches this case.

**Breaking change: non-repo agents.** The current app allows spawning agents without a
repo (runs in HOME). This plan makes all agents repo-backed. To preserve backwards compat,
the spawn form's repo field is optional — if omitted, the agent runs in HOME with no
worktree, no file tree, and no branch. It appears in the office and terminal but not in
the file explorer. This is the "legacy" mode.

**Cleanup on agent kill:**
1. Check for uncommitted changes: `git -C {worktreePath} status --porcelain`
   - If output is non-empty: keep worktree and branch. Notify client:
     "Worktree for {branchName} kept — has uncommitted changes"
   - If clean: proceed to step 2
2. Server runs `git worktree remove {worktreePath}`
   - If fails (shouldn't after step 1 check): use `git worktree remove --force`
3. Check if the branch has unpushed commits. Detect base branch:
   `git -C {repoPath} symbolic-ref refs/remotes/origin/HEAD 2>/dev/null`
   falling back to `main`, then `master`, then skip the check.
   Compare: `git -C {repoPath} log {baseBranch}..{branchName} --oneline`
   - If output is empty (no unique commits): delete branch `git branch -D {branchName}`
   - If output is non-empty: keep branch. Notify: "Branch {name} kept (has unpushed work)"
   - If no remote exists (symbolic-ref fails and no main/master): delete branch
     (local-only repo, no concept of "pushed")
4. Broadcast `worktree-cleaned` message with cleanup result

### File Tree Scanning (Hybrid State-Triggered)

**Trigger:** When `updateState()` detects a transition from WORKING to any other state
(IDLE, WAITING, MESSAGE), AND the session has a worktreePath.

**Manual refresh:** Client can send `{ "type": "refresh-tree", "sessionId": "..." }` to
trigger an immediate scan. The explorer header shows a refresh icon for this.

**Scan implementation:** Server runs in the worktree directory:
```bash
git -C {worktreePath} status --porcelain=v1
```

**Status reduction rule:** `git status --porcelain=v1` outputs two-character codes `XY`
where X = index status, Y = working tree status. Reduce to a single status for display:
- If Y is non-space (working tree has changes): use Y
- Else if X is non-space (index has changes): use X
- Priority: working tree status wins (what the user would see in `git diff`)
- `??` (untracked) → status `?`
- `!!` (ignored) → omit from tree
- Rename (`R`) → show new path with status `R`

**Tree data structure:** Parsed into a nested JSON tree:
```json
{
  "name": "myapp",
  "children": [
    { "name": "src", "children": [
      { "name": "index.js", "status": "M" },
      { "name": "new-file.js", "status": "A" }
    ]},
    { "name": "README.md", "status": null }
  ]
}
```
Files with `status: null` are tracked but unchanged. Only files with a status are
"changed files" for badge counts and conflict detection.

**WebSocket messages:**
```json
{ "type": "file-tree", "sessionId": "...", "tree": { ... }, "changedCount": 4 }
{ "type": "conflicts-update", "conflicts": [
    { "sessionIds": ["s1", "s2"], "files": ["src/config.ts"] }
  ]
}
```

**Staleness trade-off:** The file tree is stale during active WORKING state. This is
intentional — updating mid-keystroke would be visual noise. The tree refreshes when the
agent pauses (3s timeout in state detection). The manual refresh icon handles impatient users.

### Alternatives Considered
- **Approach A: fs.watch/chokidar file watcher** — Real-time but adds dependency, macOS
  edge cases with fs.watch reliability, and visual noise during active work.
- **Approach B: Polling via REST endpoint** — Simple but wasteful (constant requests
  even when nothing changed) and adds latency.
- **Approach C (chosen): State-triggered scanning** — Zero new dependencies, piggybacks
  on existing infrastructure, and natural UX timing.

### Inline Diff Viewer

**Trigger:** Click on a changed file in the explorer tree (any file with status M/A/D/?/R).

**Display:** Expands inline below the filename (accordion style). Multiple diffs CAN be
open simultaneously (not a single-open accordion — more like GitHub's file diff list).

**Data source by status:**
- `M` (modified): `git -C {worktreePath} diff -- {filePath}`
- `A` (added/staged): `git -C {worktreePath} diff --cached -- {filePath}`
  If empty (unstaged add): `git -C {worktreePath} show :{filePath}` to show full content
- `D` (deleted): `git -C {worktreePath} diff -- {filePath}` (shows deletion)
- `?` (untracked): read the raw file content from the worktree and display as all-green
  (entire file is "added")
- `R` (renamed): `git -C {worktreePath} diff -- {oldPath} {newPath}`

**Rendering:** `<pre>` block with line-level coloring:
- Green background (`#1a3a1a`) for additions (`+` lines)
- Red background (`#3a1a1a`) for deletions (`-` lines)
- No background for context lines
- Line numbers shown on left
- Max height: 300px with scroll. If diff is empty, show "No changes" placeholder.

**WebSocket messages:**
```json
{ "type": "get-diff", "sessionId": "...", "filePath": "src/index.js", "status": "M" }
{ "type": "file-diff", "sessionId": "...", "filePath": "src/index.js", "diff": "..." }
```
The `status` field tells the server which git command to use.

### Conflict Detection

**Trigger:** After every file tree scan completes, the server runs conflict detection
across ALL sessions sharing the same repoPath.

**What constitutes a conflict:** Two agents modified the same file path (conservative
heuristic — file-level, not line-level).

**Server-side:**
```javascript
function detectConflicts(repoPath) {
  const repoSessions = [...sessions.values()].filter(s => s.repoPath === repoPath);
  const conflicts = [];
  for (let i = 0; i < repoSessions.length; i++) {
    for (let j = i + 1; j < repoSessions.length; j++) {
      const pathsA = getChangedPaths(repoSessions[i]);
      const pathsB = getChangedPaths(repoSessions[j]);
      const overlap = pathsA.filter(p => pathsB.includes(p));
      if (overlap.length > 0) {
        conflicts.push({
          sessionIds: [repoSessions[i].id, repoSessions[j].id],
          files: overlap
        });
      }
    }
  }
  return conflicts;
}
```

After computing conflicts, broadcast `conflicts-update` with the full conflict list.
This replaces any prior conflict state on the client — NO separate `conflict-resolved`
message needed. The client always holds the latest conflict list from the last broadcast.
When conflicts disappear (an agent reverts a file), the next scan produces a
`conflicts-update` with an empty or reduced list, which the client applies.

**Deduplication:** The nested loop with `i < j` ensures each pair is compared once.
Broadcasting the full conflict list (not deltas) means no stale conflicts.

**UI:** Yellow warning triangle (⚠) next to conflicting files in the explorer. Tooltip
shows: "Also modified by {other-agent-name}". In the pixel office, conflicting agents
get a small yellow warning badge below their file-change badge.

### Keyboard Shortcuts

**Implementation:** Global `document.addEventListener('keydown', ...)` handler in
`shortcuts.js`. Uses `e.metaKey` (Cmd on Mac) for all shortcuts.

**xterm.js integration:** When creating each terminal instance (in `handleSessionCreated`),

```javascript
term.attachCustomKeyEventHandler((event) => {
  if (event.metaKey && ['1','2','3','4','5','6','7','8','9','e','n'].includes(event.key)) {
    return false; // Don't let xterm handle it — propagate to global handler
  }

```
This must be called at terminal creation time (inside `handleSessionCreated`), not at

### Config Persistence

**File:** `~/.agent-007/config.json`
**Schema:**
```json
{
  "repos": [
    { "path": "/Users/bill/myapp", "addedAt": "2026-03-24T12:00:00Z" },
    { "path": "/Users/bill/other-repo", "addedAt": "2026-03-24T12:05:00Z" }
  ]

```
Display slug is derived from `path` at runtime (basename), not stored.

**Behavior:**
- On server start: read config, populate file explorer with repo sections (no agents yet)
- When user adds a repo (via spawn form or "+ Add Repo"): append to config, write to disk
- When user removes a repo: remove from config, write to disk
- If file is missing on first run: create with `{ "repos": [] }`
- If file is corrupted (invalid JSON): log warning, start with empty config, do not
  overwrite the corrupted file (user may want to recover it)
- If a saved repo path no longer exists on disk: show grayed-out section with
  "Directory not found" message. User can remove it.

### Three-Way Sync Model

Source of truth: server `sessions` Map.

All sync is client-local via `switchToSession(sessionId)`:
- **Explorer → others:** Click branch name → calls `switchToSession()` locally →
  terminal shows that agent's tab, office highlights that agent
- **Office → others:** Click agent in canvas → calls `switchToSession()` locally →
  terminal switches, explorer highlights matching branch
- **Terminal tab → others:** Click tab → calls `switchToSession()` locally →
  office highlights, explorer highlights matching branch

All three converge on `activeSessionId`. The explorer highlights the branch whose
session matches activeSessionId. The office highlights the agent whose session matches
activeSessionId. The terminal shows the tab for activeSessionId.

**Multi-tab behavior:** `activeSessionId` is per-browser-tab (client-local state). Each
tab independently tracks which agent is selected. Server broadcasts (state changes, file
trees, conflicts) go to all tabs. Selection state does NOT sync across tabs — this is
intentional, matching the behavior of any IDE where each window can have a different
file open.

## Reviewer Concerns

Issues identified in round 3 adversarial review (quality score: 7/10). Addressable during
implementation:

1. **`status: null` files ambiguity.** The tree structure shows unchanged files but
   `git status --porcelain` only outputs changed files. Resolution: the file explorer
   should ONLY show changed files (it's a change explorer, not a full directory browser).
   Remove `status: null` entries from the tree structure. If a full tree is wanted later,
   add a separate `git ls-files` call in a follow-up.

2. **Dual "slug" naming.** `repoSlug` (display name) vs worktree directory component
   (`myapp-a3f2`) use the same word but mean different things. Resolution: rename the
   worktree directory component to `repoDirName` in code. Keep `repoSlug` for display only.

3. **`getChangedPaths()` undefined.** The conflict detection pseudocode calls this helper
   without defining it. Resolution: returns the flat list of paths from `session.fileTree`
   where `status !== null`.

4. **Renamed file diff command.** `git diff -- {oldPath} {newPath}` is invalid for renames.
   Resolution: use `git diff HEAD -- {newPath}` for renamed files.

5. **Spawn form as repo-addition path.** When a user fills in the repo field in the spawn
   form with a new repo, it should silently add that repo to config.json. Document this
   side effect explicitly.

6. **CDN script load order.** CDN `<script>` tags for xterm.js and FitAddon must appear
   before `<script type="module" src="app.js">` in the HTML for globals to be available.
